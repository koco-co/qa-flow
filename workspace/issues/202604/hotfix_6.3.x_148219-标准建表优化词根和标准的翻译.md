## 数据模型

### 规范建表 > 建表

#### 字段中文名解析时的词根翻译逻辑

##### 【148219】验证字段中文名解析仅触发整词精确匹配词根，不再进行分词拼接匹配

> 前置条件

1、环境要求：测试环境，已启用数据标准模块和数据模型模块（规范建表功能可访问）。

2、数据准备：

步骤一：向 `metadata_standard_root` 表写入以下两条词根记录（整词精确匹配用）：

```sql
-- 词根1：中文名为"客户"，对应精确匹配场景
INSERT INTO metadata_standard_root (root_cn, root_abbreviation, root_full_name, is_deleted, tenant_id)
VALUES ('客户', 'cust', 'customer', 0, <目标租户 tenant_id>);

-- 词根2：中文名为"编号"，对应精确匹配场景
INSERT INTO metadata_standard_root (root_cn, root_abbreviation, root_full_name, is_deleted, tenant_id)
VALUES ('编号', 'no', 'number', 0, <目标租户 tenant_id>);

-- 说明：在 6.3 之前版本，输入"客户编号"会被分词为["客户","编号"]，
-- 分别匹配到两条词根，拼接结果为缩写 cust_no，全称 customer_number。
-- 6.3 修复后，"客户编号"不存在整词词根，直接返回 null，不产生翻译结果。
```

步骤二：确认 `metadata_standard_root` 表中**不存在**中文名为"客户编号"的词根记录（用于验证整词匹配失败的路径）：

```sql
SELECT * FROM metadata_standard_root
WHERE root_cn = '客户编号' AND is_deleted = 0;
-- 预期返回空结果，若有则删除或使用其他不存在整词词根的组合词汇
```

步骤三：确认 `metadata_standard_root` 表中存在中文名为"客户"的整词词根（用于验证整词匹配成功的路径）：

```sql
SELECT root_cn, root_abbreviation, root_full_name
FROM metadata_standard_root
WHERE root_cn = '客户' AND is_deleted = 0;
-- 预期返回步骤一插入的词根，root_abbreviation = 'cust'
```

3、账号要求：测试租户管理员账号（需具备数据模型「规范建表 > 建表」访问权限，权限码 `DATAMODEL_CREATETABLE_CREATETABLE`）。

> 变更分析（供审阅参考）

- 修复文件：`StandardRootService.java`，方法 `matchRoot(String search)`
- 变更类型：业务逻辑裁剪（删除 else 分词分支）
- 触发路径：前端建表编辑页调用 `nameParseColumn` 接口，后者在标准精确匹配失败时调用 `standardRootService.matchRoot(columnNameCn)`；6.3 修复后该方法在精确匹配失败时直接返回 `null`，不再触发 `wordAnalysisUtil.parse()` 分词
- 步骤 5 和步骤 7 是核心验证点，验证分词拼接路径已被关闭
- 步骤 4 和步骤 6 验证精确匹配路径不受影响（回归保障）
- 步骤 8 验证词根管理页面本身未受影响

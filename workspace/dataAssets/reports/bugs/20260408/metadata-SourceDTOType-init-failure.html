问题分析：

metadata 的业务入口本身只是调用公共转换器，真实炸点在 datasourcex 的 SourceDTOType 类装载。两个上层仓库仍锁定 6.3.3.6-SNAPSHOT，而当前 datasourcex release_6.3.x 已到 6.3.3.8-SNAPSHOT，版本错位会让运行时出现旧 client、旧 common 或旧 sdk core 混装，从而在类初始化阶段直接失败。

环境证据： assets 仓库没有 realTimeDbList 的直接实现；queryCreateTableSql 也是 assets 调 metadata SDK 代理。说明 assets 不是根因，只是部分场景的转发层。

修复建议

以下建议基于 AI 分析，仅供参考。

P1 高  统一 datasourcex 依赖版本并重建

位置： dt-center-metadata/pom.xml, dt-center-assets/pom.xml, datasourcex release_6.3.x

修改： 将 metadata 和 assets 中的 datasourcex-client 升级到与 datasourcex 当前分支一致的版本，或统一改为同一 BOM/同一 revision 后重新构建部署。

建议至少对齐到 datasourcex release_6.3.x 当前 revision 对应版本，并确保 metadata/assets 使用同一套产物。

原因： 当前两个上层仓库固定 6.3.3.6-SNAPSHOT，而当前 datasourcex 分支已经到 6.3.3.8-SNAPSHOT，属于高风险版本错位。
P2 中  清理部署包中的旧 datasourcex jar

位置： metadata/assets 部署产物 lib 目录和构建缓存

修改： 确认最终包里只有一组 datasourcex-client、datasourcex-common、dt-insight-sdk-core；删除旧 SNAPSHOT 缓存后重新打包部署。

如果是通过制品库拉取 SNAPSHOT，需强制刷新依赖并排查服务器上是否残留旧 jar。

原因： NoClassDefFoundError 的典型根因是同名 jar 混装或旧 SNAPSHOT 未刷新，仅同步源码不会修复已部署依赖。
P3 低  为 metadata 补充依赖异常兜底日志

位置： DataTableService.initCreateTableSql, DataDbService.realTimeDbList 或 MetadataTemplate

修改： 对 NoClassDefFoundError 和 ExceptionInInitializerError 增加明确日志并返回可读错误，例如提示 datasourcex 依赖未对齐。

当前只对 DtLoaderException 做了业务化处理，依赖初始化异常会直接抛成 500。

原因： 即使根因在依赖层，接口也应输出可定位的信息，避免排查时只能看到通用 500。

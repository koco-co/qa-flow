## 数据质量

### 任务实例查询

#### 查看脏数据明细

##### 【149177】验证 Hive/SparkThrift 数据源自定义 SQL 规则查看脏数据明细时不触发全表 shuffle，返回数据量不超过 1000 条

> 前置条件

```
1. 已存在可用的 Hive 数据源和 SparkThrift2.x 数据源，且两个数据源下各有一张数据量 > 1000 行的测试表
2. 当前质量项目已开启脏数据存储（项目管理 > 脏数据管理 > 独立存储 开关为"开启"状态）
3. 脏数据管理列表中，上述两个数据源对应的"脏数据存储"开关已开启
```

> 用例步骤

**场景一：Hive 数据源**

| 编号 | 步骤 | 预期 |
| ---- | ---- | ---- |
| 1 | 进入【数据质量 > 规则任务配置】，点击某规则集进入配置页，新建一条自定义 SQL 规则:<br>- 数据源: 选择已有 Hive 数据源<br>- 数据库: 选择测试表所在数据库<br>- 数据表: 选择数据量 > 1000 行的测试表<br>- 规则类型: 自定义SQL<br>- 校验 SQL: `SELECT * FROM {table} WHERE 1=1`（确保结果行数 > 1000）<br>保存规则后立即执行该规则集 | 规则集提交执行成功，页面提示"操作成功，稍后可在任务查询中查看详情" |
| 2 | 进入【数据质量 > 任务实例查询】，找到刚执行的任务实例，点击展开查看规则详情 | 任务执行完成，规则状态显示"校验未通过" |
| 3 | 在规则行找到"查看明细"按钮，点击 | 弹出明细数据抽屉，**明细数据条数 ≤ 1000 条**，页面正常展示，无超时或报错 |
| 4 | 登录对应 Hive/Spark 集群，查询本次任务执行期间产生的 shuffle 数据量（或通过 YARN 作业历史查看 shuffle write 字节数） | shuffle write 数据量远小于历史未修复版本（修复前产生 300+ GB），任务执行时间明显缩短 |

**场景二：SparkThrift2.x 数据源**

| 编号 | 步骤 | 预期 |
| ---- | ---- | ---- |
| 5 | 重复步骤 1，数据源改为 SparkThrift2.x 数据源，其余配置相同，提交执行 | 规则集提交执行成功 |
| 6 | 进入【数据质量 > 任务实例查询】，找到对应任务实例，展开规则详情 | 任务执行完成，规则状态显示"校验未通过" |
| 7 | 点击规则行的"查看明细"按钮 | 弹出明细数据抽屉，**明细数据条数 ≤ 1000 条**，页面正常展示，无超时或报错 |
| 8 | 查询 Spark 集群本次作业的 shuffle write 数据量 | shuffle write 数据量显著低于修复前水平，磁盘占用正常 |

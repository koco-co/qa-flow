# Phase 4.5 — `tools/dtstack-cli` → `tools/dtstack-sdk` Rename Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Rename the `tools/dtstack-cli/` workspace package to `tools/dtstack-sdk/` so the npm name reflects reality (SDK is the primary export, CLI is a thin wrapper) per spec §3.2 — while preserving the user-facing `dtstack-cli` shell command verbatim so user muscle memory is untouched.

**Architecture:** Mechanical mass-rename across ≤ 6 files: directory move via `git mv`, npm package name update, root `workspaces` array entry, 4 import statements in 2 workspace test-helper files, 2 doc references. The `bin` field intentionally **stays** `"dtstack-cli"` because npm package name ≠ shell command name. Lockfile regenerates via `bun install`.

**Tech Stack:** Bun (workspaces + lockfile), npm package name semantics, no runtime code changes.

---

## Background

### Why rename

Per spec §3.2 — `package.json` already exposes SDK semantics (`"main": "src/index.ts"` + `exports` for `./` and `./adapters/playwright`); 90% of consumers are Playwright tests doing `import { sdk } from "dtstack-cli"`; only 10% is users typing `dtstack-cli xxx` at the shell. The current package name **misleads** AI and humans into thinking it's a CLI-first project.

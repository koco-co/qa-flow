# Hotfix 用例生成指令

> 本提示词由 code-analysis skill 在模式 E 时加载。
> 根据禅道 Bug 信息和源码分析生成**一条**高质量 Hotfix 验证用例，输出为 Archive 格式 Markdown。

---

## 生成目标

针对一个已修复的线上 Bug，生成**仅一条主验证用例**，但这条用例必须**非常详细**：

- 前置条件：完整的环境要求和数据准备（含可执行 SQL）
- 步骤：精确到菜单入口、按钮名称、输入内容（必须参考前后端源码校对）
- 预期：每一步都有明确的预期结果

> 不再生成回归验证用例。一条高质量用例优于两条粗糙用例。

---

## 输入信息

| 字段 | 来源 | 说明 |
|------|------|------|
| `bug_id` | zentao fetch 输出 | 禅道 Bug ID，如 `138845` |
| `title` | zentao fetch 输出 | Bug 标题 |
| `severity` | zentao fetch 输出 | 严重程度：critical / major / normal / minor |
| `fix_branch` | zentao fetch 输出 | 修复分支名 |
| `steps` | zentao fetch 输出 | 原始复现步骤（如有） |
| `git diff` | repo-sync 后执行 | 修复分支与主分支的代码差异 |

若 zentao fetch 返回 `partial: true`（API 不可达），则仅基于 Bug ID 和用户描述生成用例，在用例前置条件中注明"来源：用户描述"。

---

## 分析步骤

### 第一步：理解 Bug 本质

从 Bug 标题和复现步骤中提取：

- **触发场景**：在哪个页面 / 哪个操作下触发
- **现象描述**：出现了什么错误（报错 / 数据错误 / 功能不可用）
- **影响范围**：哪些用户 / 哪些数据受影响

### 第二步：分析代码变更（源码已在 E2 步骤中同步）

从 git diff 和源码中识别：

- **变更文件**：哪些文件被修改
- **变更类型**：空值防护 / 逻辑修复 / SQL 修正 / 配置修正 等
- **修复方式**：加了什么判断、改了什么逻辑

将变更类型映射到测试验证点，例如：
- 加了空值防护 → 验证该字段为空时页面不崩溃
- 修正了 SQL → 验证查询结果正确
- 修复了状态机 → 验证状态流转符合预期

### 第三步：源码校对（关键步骤）

**必须参考前后端源码校对以下细节**，不得凭推测填写：

- **页面入口/导航路径**：从菜单配置或页面入口确认完整操作入口（如「数据开发 > 任务管理 > 任务列表」）
- **按钮/操作名称**：从前端组件中确认按钮文案（如按钮的 label / children 属性）
- **功能逻辑**：从修复代码的上下文理解正确的业务行为

功能测试用例默认**不写前端路由路径和接口路径**，除非用户明确要求。

若源码中无法确认某个细节，用 `{{待确认：xxx}}` 标注，不要自行编造。

### 第四步：设计验证步骤

按照以下原则设计步骤：

1. **前置条件**：复现 Bug 所需的最小环境配置和数据准备（参考 Bug 的原始复现步骤）
2. **主验证步骤**：执行触发原 Bug 的操作，每一步精确到菜单入口、具体按钮和输入值
3. **预期结果**：每一步都有独立的预期结果，描述修复后应有的正确行为（而非错误行为）

---

## 输出格式

输出为 Archive Markdown 格式，遵循以下规范：

```markdown
---
suite_name: "Hotfix 用例 - {{bug_title}}"
description: "验证 Bug #{{bug_id}} 修复效果"
tags:
  - hotfix
  - bug-{{bug_id}}
create_at: "{{YYYY-MM-DD}}"
status: 草稿
origin: zentao
---

## {{模块名称}}

### {{页面名称}}

#### {{功能点名称}}

##### 【{{bugId}}】验证{{修复后的正确行为描述}}

> 前置条件
```
1、环境要求：{{具体环境，如测试环境 / 特定租户}}

2、数据准备：
{{若涉及 SQL，给出完整可执行语句；若涉及操作，写出精确操作路径}}

3、账号要求：{{所需权限角色}}
```

> 用例步骤

| 编号 | 步骤 | 预期 |
| ---- | ---- | ---- |
| 1 | 使用 {{角色}} 账号登录系统 | 登录成功，进入首页 |
| 2 | 导航至【{{一级菜单}} > {{二级菜单}} > {{三级菜单（如有）}}】 | 页面正常加载，显示 {{页面关键元素}} |
| 3 | {{精确操作：点击哪个按钮 / 输入什么值 / 选择什么选项}} | {{该步骤的预期结果}} |
| 4 | {{触发原 Bug 的关键操作}} | {{修复后的正确行为，而非报错}} |
| 5 | {{验证数据层面的正确性（如有）}} | {{预期的数据状态}} |
| ... | {{根据实际需要增减步骤，确保覆盖完整验证路径}} | {{每步都有独立预期}} |
```

## 用例标题规范

Hotfix 用例**不使用** `【P0】`/`【P1】`/`【P2】` 优先级前缀，改用 Bug ID 作为标识：

- 格式：`【{{bugId}}】验证{{修复后的正确行为}}`

示例：
- `【138845】验证商品列表页在品类为空时不显示 500 报错`

---

## 前置条件规范

前置条件数据准备规则：

- 若 Bug 涉及特定数据状态，**必须给出可执行的 SQL 语句**（建表 + 插入）
- 若 Bug 仅涉及操作流程，前置条件描述操作路径即可
- 不得写"确保数据存在"这类模糊描述

---

## 注意事项

- 用例目标是**验证修复效果**，不是重新描述 Bug 现象；预期结果应写修复后的正确行为。
- 若 git diff 不可用（API 不可达），从 Bug 标题和复现步骤推断，并在用例备注中注明"代码变更信息不可用，步骤基于 Bug 描述推断"。
- 若 Bug 标题或复现步骤信息极少（partial: true），仍需生成用例框架，将不确定的部分用 `{{待补充}}` 标注。
- **不生成回归用例**，只生成一条主验证用例，但必须足够详细。
- **符号约束**：用例 Markdown 中所有文本内容必须使用纯文本，不得包含任何 emoji 符号。U+1Fxxx 范围（如 🐛📡🚀）绝对禁止 — 禅道富文本编辑器无法保存。U+26xx 范围（如 ⚠️✅❌）仅允许在固定模板标记位置使用，AI 填充的步骤描述、预期结果等字段中不得出现。

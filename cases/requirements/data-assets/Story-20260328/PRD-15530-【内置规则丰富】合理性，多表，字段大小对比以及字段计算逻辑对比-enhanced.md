## 2. 需求背景

本页希望在「合理性校验」下新增多表字段值计算对比能力，使校验表与多个关联表按主键关联后，可以比较跨表字段表达式，支持：

- 计算结果与另一段计算逻辑对比
- 计算结果与固定值对比

与单表页相比，本页新增了：

- 关联表所属库
- 关联表
- 关联表主键
- 校验表主键
- 多表字段范围

## 4. 页面/交互详细设计

### 4.1 配置入口

用户从 `规则集配置` 页面点击【新建规则集】进入规则编辑页，在规则内容中选择：

- 规则类型: `合理性校验`
- 规则函数: `多表字段值对比`

### 4.2 多表关联配置

本页新增关联表配置，要求：

- 至少配置 1 个关联表
- 最多支持 10 个关联表
- `校验表主键` 支持多选
- `关联表主键` 支持多选
- 悬浮提示: `根据主键关联一致的数据进行计算校验`

结合现有多表内容比对能力，联合主键顺序本身也是映射规则的一部分，测试时需要验证顺序不一致或数量不一致时的行为。

### 4.3 计算逻辑配置

支持两段计算逻辑：

- `计算逻辑配置`
- `计算逻辑配置2`

字段选择范围应覆盖：

- 当前校验表字段
- 已选关联表字段

原型中使用别名形式展示，例如：

- `表a.test_col_2 + 表b.test_col_3`

同时允许手动输入运算符与函数片段，直接拼接到 SQL。

### 4.4 对比方法

支持两种模式：

- `计算结果对比`
- `计算结果值判断`

操作符支持：

- `>`
- `>=`
- `<`
- `<=`
- `=`
- `!=`

结果值默认选择当前校验表，也允许切换到关联表。

### 4.5 字段类型约束

原型明确约束：

- 仅支持数值型字段直接参与表达式
- 对于 `hive / spark / doris` 数据源，若字段为 `string`，需要先强转为 `double`

## 5. 源码补充事实

### 5.1 已确认的真实源码事实

- 前端已有多表规则基础设施：
  - `RULE_FIELD_TYPE.TABLE_CONTENT_COMPARE`
  - `RULE_FIELD_TYPE.TABLE_ROW_COMPARE`
  - 多表关联范围 `RelationRange.MultiTable`
- 后端已有多表内容比对基础设施：
  - `IntegralityMultiTableContentConfig`
  - `relationKeys`
  - `verifyTables`
  - `tableCondition`
- 后端已有多表行数差异能力：
  - `MultiTableRowsConfig`
  - `verifySettingDTO`

### 5.2 关键结论

当前目标分支里，**尚未找到与「合理性-多表字段值对比」完全对应的前后端实现类**。也就是说：

- 多表关联和主键映射能力有现成基础设施
- 但“跨表表达式计算 + 结果对比 / 固定值判断”这部分仍更像 PRD 设计，尚未在当前分支中找到明确落点

### 5.3 需要记录的缺口与风险

- 需要确认是否已新增新的 `functionId` 与结果描述模板
- 需要确认失败明细表是否已支持解析并存储“计算逻辑涉及的字段”
- 需要确认质量报告文案中是否已支持跨表表达式展示

## 6. 影响分析

### 6.1 对规则配置的影响

本页不是简单的多表内容比对，而是“多表关联后再做表达式比较”，复杂度高于现有一致性能力：

- 既要配关联表与主键
- 又要配跨表表达式
- 还要处理不同数据源 SQL 方言

### 6.2 对历史数据与兼容性的影响

- 未发现历史数据迁移逻辑
- 更可能影响新增规则定义、执行 SQL 生成、失败明细落表和报告展示
- 跨表 cast / null / decimal 精度差异是高风险点

### 6.3 对测试执行的影响

如果前端 / 后端当前分支仍未完成落地，那么测试需要优先做：

- UI 字段冻结确认
- SQL 语法兼容性确认
- 明细表结构确认

## 7. 测试关注点

### 7.1 推荐测试数据准备

建议使用 `Doris 2.x` 数据源准备一主两关联三张表：

```sql
CREATE TABLE qa_data_quality.fact_order (
  biz_id        BIGINT,
  order_amount  DECIMAL(18,2),
  discount_amt  DECIMAL(18,2),
  pay_amount    DECIMAL(18,2),
  dt            DATE
);

CREATE TABLE qa_data_quality.dim_discount_rule (
  biz_id            BIGINT,
  standard_discount DECIMAL(18,2),
  dt                DATE
);

CREATE TABLE qa_data_quality.dim_pay_rule (
  biz_id         BIGINT,
  standard_pay   DECIMAL(18,2),
  dt             DATE
);
```

推荐准备的表达式示例：

- `表a.order_amount - 表a.discount_amt = 表b.standard_discount`
- `表a.pay_amount > 1`

### 7.2 核心测试点

- 单关联表 / 多关联表
- 联合主键匹配
- 计算结果对比
- 计算结果值判断
- 当前表字段与关联表字段混合拼接
- string 字段强转后参与计算
- 失败明细是否保留主键字段、计算字段并高亮校验字段

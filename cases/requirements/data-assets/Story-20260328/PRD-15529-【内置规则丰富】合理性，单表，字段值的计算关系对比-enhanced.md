## 2. 需求背景

本页希望在「合理性校验」下新增单表字段计算关系对比能力，让测试 / 业务可以通过表达式方式校验字段间的计算逻辑是否符合预期，例如：

- `字段A > 字段B + 字段C`
- `字段B + 字段C < 固定值`

这类规则与当前分支中已经存在的「合理性-数据变化趋势 / 单调性」并不相同，属于新增能力。

## 3. 变更范围

### 3.1 规则集配置页

本页新增的核心交互来自蓝湖原型：

- `合理性校验-字段值计算对比`
- `过滤条件`
- `计算逻辑配置`
- `对比方法`
- `计算结果与字段对比`
- `计算结果值判断`

### 3.2 质量报告 / 明细

执行结果应覆盖：

- 质量报告中展示 `字段值计算对比`
- 失败时支持 `查看详情`
- 明细保留全部字段，校验字段标红
- 校验通过时不记录明细

## 4. 页面/交互详细设计

### 4.1 规则配置

用户在 `规则集配置` 页面点击【新建规则集】后，在规则内容区域选择：

- 规则类型: `合理性校验`
- 规则函数: `字段值计算对比`

随后页面需要展示：

- 字段
- 过滤条件
- 计算逻辑配置
- 对比方法

### 4.2 计算逻辑配置

`计算逻辑配置` 为表达式编辑区，要求：

- 支持点击字段快速填充到编辑框
- 字段范围仅限当前校验表
- 支持手动输入计算符号与函数片段，最终直接拼接到 SQL

示例表达式：

- `test_col_2 + test_col_3`
- `cast(test_str_col as double) * test_col_4`

### 4.3 对比方法

本页至少支持两种方式：

- `计算结果与字段对比`
- `计算结果值判断`

比较操作符支持：

- `>`
- `>=`
- `<`
- `<=`
- `=`
- `!=`

### 4.4 字段类型限制

蓝湖原型已明确：

- 仅支持数值型字段直接参与比较
- 对于 `hive / spark / doris` 类型数据源，若字段为 `string`，需在计算逻辑中显式强转为 `double`

因此前端字段选择列表、后端表达式解析、用例前置数据都必须体现这一约束。

### 4.5 结果展示

报告与详情设计遵循：

- `校验通过` 不记录明细
- `校验失败` 记录不符合计算逻辑的数据
- 质量报告文案需能反映实际表达式，例如：
  - `test_col_2 + test_col_3 > test_col_1`
  - `test_col_2 + test_col_3 > 1`

## 5. 源码补充事实

### 5.1 已确认的真实源码事实

- 前端当前分支已有 `合理性校验` 类型定义
- 前端规则说明页 `ruleDescription/index.tsx` 中，合理性仍按 `sortColumn + verifyMethod` 渲染，语义更接近“单调性 / 数据变化趋势”
- 后端当前分支存在：
  - `ReasonableDataChangeTrendConfig`
  - `ReasonableDataChangeTrendFunctionServiceImpl`
  - `ReasonableDataChangeTrendTemplateHandler`

这些类均指向 `合理性-数据变化趋势`，并不等价于本页要求的“字段值计算对比”。

### 5.2 结论

当前目标分支中，**尚未找到与「字段值计算对比」完全一一对应的前后端实现类或文案**。因此：

- 本页的字段、按钮、结果样式主要来自蓝湖原型
- 约束条件（数值型字段、string 强转、比较符集合）来自原型文本
- 菜单入口与规则集页面骨架沿用当前已有 `规则集配置` 页面

### 5.3 必须记录的实现缺口

以下内容需研发在联调前确认：

- 前端是否已新增 `字段值计算对比` 对应的 `functionId`
- 后端是否已新增表达式解析、SQL 生成、失败明细落表与结果描述模板
- 质量报告 `未通过原因 / 详情说明` 是否已按新表达式展示

## 6. 影响分析

### 6.1 对现有合理性能力的影响

当前分支里的合理性能力仍偏“趋势 / 单调性”。若本页需求上线，属于对合理性规则体系的新增函数，而非仅改文案。

### 6.2 对历史数据的影响

- 未看到历史数据迁移逻辑
- 更大影响点在于新增规则定义、执行 SQL、失败明细存储与质量报告展示

### 6.3 对兼容性的影响

- 若表达式允许 string 强转，则不同数据源 SQL 方言差异必须验证
- Doris / Hive / Spark 的 cast 语法、空值处理、精度处理都可能影响最终结果

## 7. 测试关注点

### 7.1 推荐测试数据准备

建议使用 `Doris 2.x` 数据源准备一张单表：

```sql
CREATE TABLE qa_data_quality.test_single_logic (
  biz_id        BIGINT,
  test_col_1    DECIMAL(18,2),
  test_col_2    DECIMAL(18,2),
  test_col_3    DECIMAL(18,2),
  test_col_4    DECIMAL(18,2),
  test_str_col  STRING,
  dt            DATE
);
```

建议插入至少两类数据：

- 满足 `test_col_2 + test_col_3 > test_col_1` 的正常数据
- 不满足表达式的失败数据

### 7.2 核心测试点

- 计算结果与字段对比
- 计算结果值判断
- 六种操作符是否全部可选
- string 字段是否要求显式强转
- 非数值型字段是否被禁止参与配置
- 失败详情是否只记录失败数据
- 质量报告是否展示真实表达式

### 7.3 需研发确认的开放问题

- 本页能力是否已经在当前分支完成开发
- 若尚未完成，前后端的最终字段名、按钮名、异常提示文案需在联调前冻结
